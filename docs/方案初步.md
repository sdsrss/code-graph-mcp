## 一、 核心技术栈选型（极致轻量化方案）

为了实现“无需配置、占用内存极小、插上就能用”，建议摒弃 Python/Node.js 生态中庞大的环境依赖，采用编译型语言开发单一二进制文件：

- **开发语言**：**Go** 或 **Rust**。编译后只有一个几 MB 的可执行文件，启动时间 < 50ms，完全契合“轻量开箱即用”。
- **统一存储核心**：**SQLite + sqlite-vec + FTS5**。
  - 无需安装任何外部数据库服务。
  - **普通表**：存储 AST 结构、知识图谱（关系边）和 Merkle Tree 缓存。
  - **sqlite-vec 扩展**：直接在 SQLite 中实现高性能的本地向量检索。
  - **FTS5 扩展**：SQLite 内置的全文检索引擎，用于精准词元（Token）匹配。
- **代码解析引擎**：**Tree-sitter**。速度极快，支持 C 语言级别绑定，原生支持多语言增量解析。
- **本地向量化（Embedding）**：**ONNX Runtime**。在 Go/Rust 内部直接集成 ONNX 运行时，内嵌一个极小的开源代码量化嵌入模型（如 30MB 的 `bge-micro` 或 `all-MiniLM-L6-v2`）。无需安装 Ollama，程序自己就能算特征向量！

## 二、 核心模块架构设计

## 1. 增量索引与 Merkle Tree 缓存模块

- **智能过滤**：系统启动时读取 `.gitignore`，自动跳过 `node_modules`、`.git`、`dist` 等目录。
- **Merkle Tree 增量判定**：为每个目录和文件计算 Hash，构建一棵 Merkle 树存入 SQLite。当 IDE 保存文件时，通过文件系统的 Watcher（监听器）捕获事件，仅在 Hash 发生变化的叶子节点（具体文件）触发重算，实现**毫秒级增量更新**。

## 2. AST 智能分块与知识图谱（Knowledge Graph）构建

普通的切分是按“每 500 个字符”硬切，这会破坏代码逻辑。你的系统将这样处理：

- 用 Tree-sitter 将代码解析为 AST，按**“函数 (Function)”、“类 (Class)”、“接口 (Interface)”**等语义节点进行智能切块。
- **提取依赖边构建图谱**：在遍历 AST 时，提取出：
  - `CallEdges`（谁调用了谁）
  - `HTTP_Routes`（例如提取出 `@Get('/api/user')` 对应的处理函数）
  - `ClassInheritance`（继承关系）
- **图谱入库**：将这些关系存入 SQLite 的关系表中（如 `Edges(source_node, target_node, relation_type)`）。

## 3. 定制特征嵌入（Custom Embeddings）与混合搜索

- **定制上下文注入**：在对某个函数进行向量化前，不只传入函数代码本身，而是拼接它的图谱信息。例如：`"路由: /api/login, 函数: authUser, 调用了: db.query。代码: ..."`。将这段组合文本送入内置的 ONNX 模型生成向量。
- **混合搜索（Hybrid Search）**：当 Claude 搜索时，同时执行：
  1. **sqlite-vec 向量搜索**（捕捉语义意图：“处理登录失败的逻辑”）
  2. **SQLite FTS5 全文搜索**（捕捉精准变量名：“authUser”）
      将两者的得分通过 RRF (Reciprocal Rank Fusion) 算法合并，得出最准确的 Top-3 代码块。

## 三、 MCP 工具接口设计（给 Claude 调用的 API）

为了最大化节省 Token 并让 Claude 瞬间理清复杂逻辑，你的 MCP Server 需要向 Claude Code 暴露以下几个极为智能的 Tools：

1. `semantic_code_search(query: string)`
   - **作用**：结合图谱和向量的混合搜索。
   - **返回**：不返回几千行的长文件，只返回最高相关的 3 个具体函数（AST Chunk）及它们所在的文件路径。
2. `get_call_graph(function_name: string, depth: int = 2)`
   - **作用**：这是节省 Token 的杀手锏！利用 SQLite 的 **递归 CTE (WITH RECURSIVE)** 查询语句，在 10 毫秒内查出某个函数的完整上下游调用链。
   - **返回**：精简的依赖树文本（如：`A() -> B() -> C()`），Claude 无需阅读 B 和 C 的源码就能懂逻辑。
3. `find_http_route(route_path: string)`
   - **作用**：让 Claude 根据前端请求（如 `/api/v1/users`）直接跳转到后端的 Controller 函数。
4. `get_ast_node(file_path: string, symbol_name: string)`
   - **作用**：按需精准获取某个特定类或函数的源码，绝不多给一行废话。

## 四、 为什么这套架构能“完胜”传统方案？

1. **极度省 Token 且防幻觉**：面对“A 调用 B，B 调用 C”的复杂 Bug，传统大模型需要把 A、B、C 三个文件（可能上万行）全部塞入上下文去“读”。而你的系统，Claude 只需调用 `get_call_graph("A")`，系统在 100ms 内直接返回 SQLite 中查到的图谱关系。Claude 瞬间理解架构，只取有 Bug 的那个函数。
2. **绝对隐私与零配置**：编译成单个二进制文件（例如 `my-mcp-server.exe`）。用户只需在 Claude Code 的配置中写一行：`"command": "my-mcp-server", "args": ["--dir", "./"]`，不需要 Docker，不需要配置 Milvus，不需要 OpenAI API Key。
